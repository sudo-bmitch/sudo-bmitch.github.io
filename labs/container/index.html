What is a Container

A Container is a Process

Lets see how a container is a process, using the ps command to show processes. First, verify the before state:

ps -ef | grep tail

Now run a container in the background with the tail command which will hang indefinitely:

docker container run -d --rm --name test busybox:latest tail -f /dev/null

And check for the running processes again, both inside and outside of the container:

ps -ef | grep tail

docker container exec test ps -ef

What is different about the tail process inside the container vs on the host?

Answer

The PID value for the same process is different, showing the pid namespace effect.

That was fairly trivial, but hopefully showing a major different from VM’s.

Cleanup before the next step:

docker container kill test

Container Isolation

If the container is just a process, how well is it isolated. For that, lets start by looking at the full process list on the host:

ps -ef

Now lets run an interactive container and look at the processes inside:

docker container run -it --rm centos:latest bash
ps -ef

There’s not much running in this container. In fact pid 1 is just bash, the process we specified on the run command. Lets try to break things:

hostname
hostname hacker-wuz-here
date 123123591999
id

Looks like we’re fairly restricted, despite being root in the container. But a container is a process in the same kernel with namespaces and cgroups, lets have a look at those:

uname -v
ls -al /proc/self/ns

The uname value doesn’t look normal for a CentOS kernel. And the namespace list is just a bunch of numbers. Exit the container and compare that to the host.

exit

uname -v
ls -al /proc/self/ns

It looks like the kernel is identical to the host, and the namespace values for everything except cgroup and user are different.
